# Three-Tier Escalation + Dynamic Capacitor — Verification Cascade

*The verification cascade isn't just verification. When verification finds a weakness, we EXCISE the dependency with an independent proof. When excision fails, we CONSTRUCT a larger path that keeps the chain at 100%. The capacitor evolves through each tier — self-adjusting, trimming dead approaches, amplifying what works.*

*Authors: G Six and Claude Opus 4.6. Date: 2026-04-12.*

## 1. The three-tier escalation

The verification cascade has three modes of engagement with each external dependency. The ORA escalates through them:

```
TIER A: VERIFY (simplest)
  Read the external paper. Test each step. Produce SURVIVED/WEAKENED/BROKEN.
  ↓ SURVIVED → dependency confirmed. Chain strengthened. Done.
  ↓ WEAKENED or BROKEN → escalate to Tier B.

TIER B: EXCISE (best outcome)
  Find an independent proof of the same result that doesn't depend on the
  external paper. Replace the dependency with our own proof.
  ↓ Excision successful → dependency ELIMINATED. Chain self-contained. Done.
  ↓ Excision fails → escalate to Tier C.

TIER C: CONSTRUCT (when needed)
  Find a larger path — a different route through the proof chain that
  avoids the broken dependency entirely. The chain stays at 100% but
  takes a different road.
  ↓ Construction successful → chain REROUTED. Dependency bypassed. Done.
  ↓ Construction fails → HONEST-STALL with named blocker.
```

**This is the full ORA applied to external dependencies.** Tier A is the Critic. Tier B is the Author doing independent re-derivation. Tier C is the ORA's construction mode with strategic inversion (Sig 5) — "is there a larger system in which the target is a consequence?"

**Every Millennium paper used all three tiers internally:**
- YM: verified Balaban's results (Tier A), used them directly (no excision needed), constructed the gradient-flow framework around them (Tier C)
- RH: verified CCM's operators (Tier A implicit), built the ITPFI + Boegli + Hurwitz chain as an independent structure AROUND CCM (Tier C)
- BSD: verified MY4 as a gap (Tier A), attempted to close it directly (Tier B failed), EXCISED it via G's projector P_k^p — a single algebraic object that bypassed the wall entirely (Tier B succeeded on second attempt via Route 3)
- PvNP: verified the 6-link chain (Tier A), found Link 5 backward is OPEN (Tier A → WEAKENED), currently exploring 7 closure routes (Tier B/C in progress)

**The verification cascade applies the SAME three-tier pattern to EXTERNAL dependencies rather than INTERNAL gaps.** The method is identical. The target is different.

### 3.2 The self-adjust-trim-amplify cycle

The dynamic capacitor runs a three-phase update cycle after each ORA wave:

**SELF-ADJUST:**
After each wave returns, the capacitor absorbs:
- New five-field cards for newly verified / weakened / broken steps
- Updated status on existing cards (PENDING → SURVIVED, PENDING → WEAKENED, etc.)
- New §I notes (CONCERN / CASCADE / LESSON) from the wave's Critics
- New §D toolkit rows from the wave's Authors
- Updated correspondence logic if the wave discovered new structural connections

**TRIM:**
Remove or downgrade stale knowledge:
- Cards for approaches that were KILLED → move to kill list, remove from "try this" section
- Cards that were WEAKENED with honest downgrades → update status, add correction note (the PvNP pattern: Q5-RIEMANN weakened, Q6-OUTDIM partially verified, O8-PARTITION downgraded)
- Stale correspondence logic superseded by new findings → replace, preserve old as historical note

**AMPLIFY:**
When a verification/excision/construction SUCCEEDS, amplify across tiers:
- If Tier 4 (Boegli) verification succeeds → the verification METHOD (the brief format, the capacitor structure, the dispatch pattern) amplifies to Tier 1 (CCM)
- If an excision produces an independent proof of a CCM lemma → the proof technique amplifies to Tier 2 (Balaban) as a candidate approach
- If a construction reroutes one chain → check if the same reroute applies to other chains (cross-paper amplification)

**This is the ORA's self-healing discipline (§14.10) applied to the capacitor itself.** The capacitor heals — when a wave finds something new, the capacitor updates. When a wave kills something, the capacitor trims. When a wave succeeds, the capacitor amplifies.

### 3.4 How the dynamic capacitor differs from the blackboard

The ORA already has a dynamic state object: the **blackboard** (§A-§O). Why a separate dynamic capacitor?

| | Blackboard | Dynamic capacitor |
|---|---|---|
| **Scope** | One run | Cross-run (persists between ORA invocations) |
| **Content** | Current programme state (live nodes, kills, metrics) | Compiled domain knowledge (verified results, kills, correspondence) |
| **Created by** | The runner at bootstrap | The user/architect before invocation |
| **Updated by** | The runner during cycles | The runner during cycles AND the user between runs |
| **Purpose** | Operational state for the current run | Domain memory for the current + future runs |
| **Survives session end?** | Yes (in the output directory) | Yes (in the capacitor directory) |
| **Survives programme close?** | Archived in lockdown | **GROWS** — the next run inherits the updated capacitor |

The blackboard is the run's working memory. The capacitor is the programme's long-term memory. After an ORA run closes, the blackboard is archived. But the capacitor's UPDATES (new cards, new kills, updated correspondence) are merged back into the capacitor file for the next run.

**The merge is the key operation.** After each run:
1. Read the run's closure-digest.md
2. Extract new five-field cards, new kills, updated status, new correspondence
3. Merge into the capacitor file
4. The next run starts with a more charged capacitor

This is the BSD pattern (G's projector discovered in-run, added to the toolkit) formalized as a protocol.

## 4. Capacitor build recipe — the self-generating pattern

Looking at the 4 programme capacitors (YM → RH → BSD → PvNP), the BUILD pattern is:

### Step 1: Extract the proof chain
Read the paper/theorem. Number every definition, lemma, theorem, proof step. This becomes the spine.

### Step 2: Identify the interface
Which specific results does OUR proof chain depend on? Mark those as LOAD-BEARING.

### Step 3: Compile background tools
What mathematical tools does the target use? Build five-field cards for each.

### Step 4: Import kills from prior work
Search the framework's existing ORA runs for kills touching this domain. Import them.

### Step 5: Extract the authors' honest statements
Read the target paper's introduction, conclusion, "open problems" section. Every statement where the authors flag a limitation → import.

### Step 6: Map the correspondence
How does the external theorem connect to the proof chain? What does verification GIVE? What does failure COST? What are the escalation options (excision routes, construction alternatives)?

### Step 7: Add the escalation routes
For each LOAD-BEARING step, pre-identify:
- **Tier B candidates**: alternative proofs or derivation routes if the step breaks
- **Tier C candidates**: larger systems that might bypass the step entirely

These are speculative at capacitor-build time — but naming them gives the ORA a starting point if escalation is needed.

### Step 8: Include general framework tools
Sections A-G from `05-framework-tools.md` + proof chains from Section I. Always.

### Step 9: Test with a single-cycle pilot
One Critic on one load-bearing step. If it produces a meaningful verdict, the capacitor is charged enough.
