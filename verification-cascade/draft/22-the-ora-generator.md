## The machine that builds memory

The ORA generator is a meta-level tool: an ORA run whose output is a CAPACITOR for a subsequent ORA run. It takes a proof chain (or an external paper) as input and produces a fully charged capacitor as output — correspondence tables filled, Six Patterns applied, grammar analyzed, operations mapped, kills imported, escalation routes pre-identified.

Without the generator, building a capacitor is manual: read the paper, extract the chain, build the tables by hand. The generator automates this — systematically, consistently, and with the full creative substrate of the Seven Keystone Files applied to every proof step.

## The 10-step process

The generator follows a fixed 10-step procedure:

**Step 1 — Acquire the target.** Download the paper (WebFetch from arXiv if given an ID) or read the file. Extract and number every definition, axiom, lemma, theorem, and proof step in logical dependency order. Identify load-bearing steps (marked with *). Every step starts as PENDING VERIFICATION for external papers, or at its current status for framework proof chains.

**Step 2 — Build the correspondence tables.** For each load-bearing concept, identify its image in four domains (spectral, geometric, algebraic, information-theoretic). Use the framework's existing correspondences as templates. Mark empty cells as "NO KNOWN IMAGE — candidate for discovery." Activate Tier 2 domains (the 37 R-Theorem families) based on the target's mathematical content.

**Step 3 — Apply the Six Patterns.** For each load-bearing step, test all six patterns (P1 geometric reinterpretation, P2 holonomy, P3 scale-setting, P4 topological rigidity, P5 zeta regularization, P6 projection diagnosis). Identify the highest-leverage pattern per step. In verification mode: patterns identify vulnerabilities. In excision mode: patterns suggest alternative proofs. In construction mode: patterns guide the search for larger systems.

**Step 4 — Apply the predictive grammar.** For each formula or numerical claim, tag the grammar rule it follows (SUM, PRODUCT, YUKAWA, EXPONENTIAL, LOG, FRACTIONAL, RATIO, DIFFERENCE). Deviations from the grammar are findings — either new rules or errors. The grammar constrains the search space for excision and construction.

**Step 5 — Build the operations equivalence table.** For each mathematical operation in the proof chain (inner product, tensor product, quotient, limit, duality, gap, trace, commutator, projection), identify its equivalent in each active domain. Empty cells are research targets.

**Step 6 — Identify the interface with the framework.** Read the four framework proof chains. For each load-bearing step in the target, identify: which framework step depends on it, what happens if it breaks, whether alternative routes exist.

**Step 7 — Build the kill list seed.** Import kills from prior ORA runs touching this domain. Search the H4 closure corrections, the P vs NP toolkit kills, the BSD MY4 corrections, the RH honest negatives.

**Step 8 — Pre-identify escalation routes.** For each load-bearing step: Tier B candidates (alternative proofs in the literature, independent derivation routes via Six Patterns, transposition dictionary pathways) and Tier C candidates (larger systems via P1, different routes through the chain, alternative proof chains in the framework).

**Step 9 — Compile the capacitor.** Assemble everything into the dynamic capacitor format: META section (version, counts, status), general framework tools (A-G), programme-specific content (H.1-H.12), framework proof chains (I), and empty MERGE LOG.

**Step 10 — Validate with a pilot.** Dispatch one Critic on one load-bearing step using only the capacitor. If the Critic produces a meaningful verdict, the capacitor is charged enough. If not, identify what's missing and iterate.

## The tiered domain system

The generator does not dump all 39+ domains into every capacitor. It selects domains based on the target's content:

**Tier 1 — The four pillars (ALWAYS active):** Spectral, geometric, algebraic, information-theoretic. These four columns appear in every correspondence table for every capacitor. They are the minimum.

**Tier 2 — R-Theorem families (activated by target domain):** When the target paper involves operator algebras, the 6 OA R-Theorems are activated. When it involves QFT, the 9 QFT R-Theorems are activated. When it involves number theory, the analytic NT and representation theory domains are activated. The activation is based on the target's mathematical content, identified during Step 1 (proof chain extraction).

**Tier 3 — Structural domains (activated in excision/construction mode):** Category theory, homological algebra, combinatorics, topology, dynamical systems, quantum information. These are NOT in the initial correspondence table — they are AVAILABLE when an agent needs new pathways. If Tier A verification fails and the capacitor escalates to Tier B or C, the Tier 3 domains are added to the correspondence table for the escalated run.

**Tier 4 — Frontier domains (discovery targets):** The 12 unopened domains from Section 21. Finding a new correspondence in these domains IS a result. They are not pre-loaded into the capacitor — they are listed as "available for exploration" so that an agent in Tier C construction mode can try opening a new domain as a route around a broken dependency.

The tiered system keeps the capacitor focused (4-8 active domains) while making all 39+ domains accessible when needed (via escalation to Tier 3/4). This is the I-v6-5 lesson applied to domains: don't selectively include based on classification (the runner will misclassify). Include the core (Tier 1) always. Let the agent discover what else it needs.

## The meta-ORA architecture

The generator is the FIRST LAYER of a two-layer ORA system:

```
Layer 1: Capacitor-building ORA runs
  Input: target paper / proof chain
  Output: charged capacitor
  Duration: short (1 session, 1-3 waves)
  Mode: execute (capacitor-building brief)

Layer 2: Verification / excision / construction ORA runs
  Input: capacitor (from Layer 1) + verification brief
  Output: SURVIVED / WEAKENED / BROKEN verdicts + new mathematics
  Duration: variable (1 session to multi-session)
  Mode: execute with three-tier escalation
```

Layer 1 runs ONCE per target (with possible re-runs if the pilot validation fails). Layer 2 runs as many times as needed, with the capacitor growing dynamically between runs.

The same ORA prompt (`01-the-prompt.md`) drives both layers. The difference is the BRIEF: Layer 1's brief says "build a capacitor for target X." Layer 2's brief says "verify/excise/construct using capacitor Y." The ORA is the operating system. The brief is the application. The capacitor is the data.
