## Part II — The mechanism

### BLOCKED-DECOMPOSED is the existing PAC primitive that does this

From ORA v8 prompt §7 Step 4 (LOCK):

> *"If no invariant protects the result, before declaring BLOCKED, attempt decomposition: decompose the protective-invariant requirement into 2–4 sub-requirements. Re-attempt Lock on each. If decomposition succeeds, verdict is BLOCKED-DECOMPOSED and the sub-requirements become new §E nodes."*

This primitive is:
- **Recursive** — each sub-requirement can itself be BLOCKED-DECOMPOSED
- **Unbounded in depth** — the recursion continues until every leaf is LITERATURE, PROVED, CLASSICAL, or a named HONEST WALL
- **Already in the architecture** — no new primitive needs to be written

The PAC can execute this on any CONDITIONAL wall in the ring. The only change required: dispatch the PAC in DECOMPOSE mode, not BYPASS mode.

### BYPASS vs DECOMPOSE — the critical distinction

| Mode | Instruction | What the PAC does |
|---|---|---|
| **BYPASS** | "Find a structural argument that closes this gap using the capacitor" | Dispatches Authors to find SINGLE-TOOL routes (e.g., "use Borel summability to close H4"). Returns ADVANCED/BLOCKED per route. |
| **DECOMPOSE** | "Decompose this conditional into sub-requirements until every leaf is PROVED or a named HONEST WALL" | Dispatches Authors to PARTITION the claim, recurse on each sub-part, terminate when all leaves are either proved or specifically named open problems. |

Routes A/B/C/D on H4 were all BYPASS attempts. They tried single-tool bypass of H4's ~4 Appendix L sub-hypotheses. Each failed at its NATURAL WALL — where the tool stopped applying.

DECOMPOSE mode is different: instead of finding one tool that covers everything, it finds the RIGHT tool for each sub-sub-link. Multiple tools, each applied at its natural scope, chained together to produce the whole.

This is how Paper 13 (RH) works: CCM operators (tool 1) + ITPFI (tool 2) + Bögli (tool 3) + Hurwitz (tool 4). No single tool closes RH. Same for Paper 26 (BSD): BC over K + bridge family + GRH + Deuring CM + Kolyvagin. No single tool.

**The decomposition path says: stop trying to close conditionals with single-tool bypasses. Start decomposing them into chains where each sub-link uses the right local tool.**

### The dispatch instruction template

To trigger DECOMPOSE mode on a specific wall, dispatch the PAC with:

```

  [INITIAL_SUB_LINKS from Appendix L, strategy files, etc.]

  1. Attempt LITERATURE closure (cite existing theorem) → LITERATURE
  2. Attempt framework-native CONSTRUCT → PROVED
  3. Attempt transposition via capacitor → PROVED via TRANSPOSITION
  4. Otherwise DECOMPOSE into 3-6 sub-requirements and recurse

```

This is the instruction format. Substitute [WALL_NAME] and [VERTEX] per dispatch.

### Status dictionary for decomposed chains

Every sub-link at every depth receives one of these statuses:

| Status | Meaning | Closure condition |
|---|---|---|
| PROVED | Proved in framework papers | In-situ proof exists |
| LITERATURE | Proved externally, cited | Peer-reviewed theorem referenced |
| CLASSICAL | Obvious at sight | Euclid-level identity |
| QG5D | Derives from QG5D's 22 theorems | Chain to QG5D on disk |
| TRANSPOSITION | Proved via capacitor transposition | Capacitor cell cited |
| PARTIAL | Partially proved; named residue | Explicit gap description |
| OPEN | Named HONEST WALL | Specific open problem stated |
| BLOCKED-DECOMPOSED | Itself decomposes into sub-requirements | Sub-requirements listed |

A decomposition is COMPLETE when every leaf has status ∈ {PROVED, LITERATURE, CLASSICAL, QG5D, TRANSPOSITION, OPEN}. "OPEN" is allowed — it's the honest wall. "BLOCKED-DECOMPOSED" is a transient state; the recursion continues on its sub-requirements.

## Part IV — The honest caveats

### Caveat 1 — Some decompositions might not terminate cleanly

Schanuel's conjecture is the canonical hard case. Schanuel IS an external conjecture. The framework might decompose to "algebraic independence of two specific numbers" rather than "full Schanuel" — narrower but still open.

**Verdict**: decomposition will NARROW walls, not always ELIMINATE them. That's still progress.

### Caveat 2 — The total link count explodes

Current: ~199 theorems. Post-decomposition: potentially 500-800. Each individually PROVED, each small, but a LOT to write down, verify, and maintain.

**Verdict**: months of work. The decomposition path is a PROGRAMME, not a sprint. Cascade one vertex at a time.

### Caveat 3 — Not all decompositions are equal in size or difficulty

Some walls (e.g., H4) might decompose into 30 leaves, most LITERATURE. Others (e.g., Schanuel) might decompose into 50+ leaves, 20 of which are still OPEN.

**Verdict**: the decomposition discipline must be HONEST. Report "wall decomposed into N leaves, X proved, Y open with named residuals." Don't overclaim.

### Caveat 4 — Some leaves will need external closure

Even in the best case, some residual walls will be long-standing open problems (Borel summability of 4D YM, Schanuel, non-perturbative OPE for 4D gauge). These stay OPEN until the mathematical community closes them.

**Verdict**: the programme's position becomes "waiting on N specific open problems in named fields" rather than "conditional on H4/CCM/CBB as a whole." Much stronger claim structure.

### Caveat 5 — The PAC might fail to decompose cleanly in DECOMPOSE mode if the framework's vocabulary doesn't yet reach the sub-lemma level

Some conditionals might be "impossible for us to decompose" for the same reason H4 was "impossible for us to bypass" via single-tool routes — the framework's BC operator-algebraic language might not reach the specific sub-lemma that needs to be proved.

**Verdict**: the decomposition attempt is itself DIAGNOSTIC. If a wall resists decomposition, that's INFORMATION about where the framework's language is incomplete.
